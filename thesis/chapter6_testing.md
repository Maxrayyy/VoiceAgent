# 第六章 系统测试与结果分析

本章介绍系统的测试方案、测试环境和测试结果，通过单元测试、端到端测试和 RAG 检索性能评估对系统的正确性和性能进行全面验证。

## 6.1 测试环境

### 6.1.1 硬件与软件环境

系统测试在以下环境中进行：

| 项目 | 配置 |
|------|------|
| 操作系统 | Linux 6.6 (WSL2) |
| Python 版本 | 3.10 |
| 测试框架 | pytest |
| 浏览器 | Chrome |
| 网络环境 | 本地服务器 + 云端 API |

### 6.1.2 测试数据

（1）**知识库数据**：从飞机维修手册 PDF 提取的文本数据，约 432KB，经智能切分后生成若干文档片段。

（2）**评估数据集**：使用 LLM 自动生成的 45 条测试查询，覆盖航电设备、座椅系统、通信系统、水系统、维修工艺等多个飞机维修子领域。每条查询配有标准答案（golden_content），为知识库文档中的精确子串。

（3）**测试音频**：使用 CosyVoice TTS 生成的 16kHz WAV 格式测试音频文件，用于端到端测试。

## 6.3 端到端测试

### 6.3.1 测试方案

端到端测试通过 WebSocket 测试客户端（`scripts/e2e_test_client.py`）模拟前端行为，将预录制的测试音频通过 WebSocket 发送给后端，收集完整的处理结果（STT 识别 → RAG 检索 → LLM 生成 → TTS 合成）。

测试客户端支持三种测试模式：

（1）**单条查询测试**：发送单条音频，验证完整流程的正确性和延迟。

（2）**多轮对话测试**：连续发送多条音频，验证多轮对话上下文管理和指代消解功能。

（3）**打断测试**：在系统播报过程中发送新音频，验证打断机制的正确性。

### 6.3.2 测试流程

端到端测试的执行流程如下：

（1）启动后端服务，加载知识库索引。

（2）测试客户端建立 WebSocket 连接。

（3）读取测试音频文件，按 3200 字节（100ms 音频）为一个块分片发送。

（4）收集后端返回的所有消息，记录各阶段的时间戳。

（5）验证 STT 识别结果、LLM 生成内容和 TTS 音频输出的完整性和正确性。

端到端测试验证了以下功能：

## 6.4 RAG 检索性能评估

### 6.4.1 评估方法

使用自动生成的 45 条测试查询对 RAG 检索系统进行性能评估。评估采用三项标准信息检索指标：

（1）**Hit Rate@K**（命中率）：在 Top-K 检索结果中至少包含一个正确文档的查询比例。反映检索系统的基本召回能力。

$$HitRate@K = \frac{1}{|Q|} \sum_{q \in Q} \mathbb{1}[\exists d \in TopK(q): d \text{ is relevant}]$$

（2）**MRR@K**（平均倒数排名）：所有查询中第一个正确文档排名的倒数的平均值。反映正确文档的排名位置，排名越靠前 MRR 越高。

$$MRR@K = \frac{1}{|Q|} \sum_{q \in Q} \frac{1}{rank_q}$$

其中 $rank_q$ 为查询 $q$ 的第一个相关文档在 Top-K 中的排名。

（3）**nDCG@K**（归一化折损累积增益）：考虑所有相关文档排名位置的综合指标，对排名靠前的相关文档给予更高的权重。

$$nDCG@K = \frac{DCG@K}{IDCG@K}, \quad DCG@K = \sum_{i=1}^{K} \frac{rel_i}{\log_2(i+1)}$$

### 6.4.2 对比实验设计

为验证各项优化措施的效果，设计了以下四组对比实验：

| 实验编号 | 配置 | 说明 |
|---------|------|------|
| 实验 1 | baseline (dense) | 纯稠密向量检索 |
| 实验 2 | hybrid | 混合检索（稠密 + BM25 + RRF） |
| 实验 3 | hybrid + rerank | 混合检索 + 交叉编码器重排序 |
| 实验 4 | optimized hybrid + rerank | 优化版混合检索 + 重排序（含术语词典等） |

所有实验均使用 Top-K=5 的检索参数，在相同的 45 条测试查询上进行评估。

### 6.4.3 实验结果

四组实验的评估结果如表 6-1 所示。

**表 6-1 不同检索策略的评估结果对比**

| 检索策略 | Hit Rate@5 | MRR@5 | nDCG@5 |
|---------|-----------|-------|--------|
| baseline (dense) | 0.8444 | 0.6626 | 0.7071 |
| baseline (dense, 优化 Embedding) | 0.8667 | 0.7293 | 0.7601 |
| hybrid | 0.9333 | 0.8259 | 0.8537 |
| hybrid + rerank | 0.9778 | 0.9667 | 0.9678 |

### 6.4.4 结果分析

（1）**混合检索的提升效果**

从 baseline 到 hybrid 的实验结果显示，引入 BM25 稀疏检索并使用 RRF 融合后，三项指标均有显著提升：

- Hit Rate：0.8667 → 0.9333（+7.7%）
- MRR：0.7293 → 0.8259（+13.2%）
- nDCG：0.7601 → 0.8537（+12.3%）

分析原因：航空维修手册中包含大量专业术语，BM25 的关键词精确匹配能力与稠密检索的语义理解能力形成互补。例如，对于查询"ARINC629 数据总线的工作原理"，稠密检索可能匹配到描述其他通信协议的语义相似段落，而 BM25 能够精确匹配包含"ARINC629"关键词的文档片段。RRF 融合使两路检索结果中排名靠前的文档获得更高的综合评分，有效提升了检索质量。

（2）**交叉编码器重排序的提升效果**

从 hybrid 到 hybrid + rerank 的实验结果显示，重排序带来了显著的质量提升，尤其在排序相关指标上：

- Hit Rate：0.9333 → 0.9778（+4.8%）
- MRR：0.8259 → 0.9667（+17.1%）
- nDCG：0.8537 → 0.9678（+13.4%）

分析原因：MRR 的大幅提升说明重排序显著改善了正确文档的排名位置。混合检索可能将相关文档放在第 2-5 位，而重排序能够将其提升到第 1 位。交叉编码器通过对查询-文档对的联合编码，能够捕获更精细的相关性信号，从而给出更准确的排序。

（3）**最终系统性能**

经过全部优化后的系统（hybrid + rerank）在 45 条测试查询上实现了：

- **Hit Rate@5 = 97.78%**：45 条查询中有 44 条在 Top-5 结果中包含正确答案，仅 1 条未命中。
- **MRR@5 = 0.9667**：正确答案平均排名接近第 1 位，绝大多数查询的正确答案被排在首位。
- **nDCG@5 = 0.9678**：综合排序质量接近理想状态。

（4）**各阶段优化贡献**

将优化效果进行阶段性分解：

| 优化阶段 | Hit Rate 提升 | MRR 提升 | nDCG 提升 |
|---------|-------------|---------|----------|
| 引入混合检索 | +7.7% | +13.2% | +12.3% |
| 引入重排序 | +4.8% | +17.1% | +13.4% |
| 累计提升 | +12.5% | +30.3% | +25.7% |

重排序对 MRR 的贡献最大（+17.1%），说明重排序的核心价值在于将正确答案推到排名首位，直接改善用户获取正确信息的效率。

## 6.5 延迟性能分析

### 6.5.1 各模块延迟分解

通过端到端测试客户端记录的时间戳数据，对系统各模块的延迟进行了分析：

| 处理阶段 | 典型延迟 | 说明 |
|---------|---------|------|
| STT 实时识别 | ~200ms | 边说边识别，延迟为最后一段音频到最终结果 |
| 查询改写 | 100-300ms | Qwen-turbo 单次调用 |
| RAG 混合检索 | 200-500ms | FAISS + BM25 + RRF 融合 |
| 重排序 | 100-300ms | gte-rerank API 调用 |
| LLM 首 token | 300-800ms | Qwen-plus 流式推理启动 |
| TTS 首帧音频 | 200-500ms | CosyVoice 合成启动 |

### 6.5.2 端到端延迟

| 指标 | 串行模式（估计） | 流式模式（实测） |
|------|--------------|--------------|
| 用户说完到首帧音频 | ~5.6s | ~1.6s |
| 延迟降低比例 | - | 约 71% |

流式模式下各阶段高度并行，LLM 生成的第一句话完成后 TTS 即开始合成，用户在 1-2 秒内即可听到第一段回答。

## 6.6 本章小结

本章通过单元测试、端到端测试和 RAG 检索性能评估对系统进行了全面测试。单元测试验证了评估指标、BM25 索引、查询改写、元数据处理和 STT 配置五个模块的正确性，全部 16 个测试用例通过。端到端测试验证了从语音输入到语音输出的完整流程的功能正确性。RAG 检索评估通过四组对比实验定量验证了各项优化措施的效果，最终系统实现了 97.78% 的命中率、0.9667 的 MRR 和 0.9678 的 nDCG。延迟分析表明流式架构将端到端延迟从约 5.6 秒降低至约 1.6 秒。
