# 第二章 相关技术

本章介绍系统设计与实现过程中涉及的关键技术，为后续章节的系统设计和实现提供技术基础。

## 2.1 检索增强生成（RAG）

### 2.1.1 RAG 基本原理

检索增强生成（Retrieval-Augmented Generation, RAG）是一种将信息检索与文本生成相结合的技术框架。其核心思想是：在大语言模型生成回答之前，先从外部知识库中检索与用户问题相关的文档片段，将这些片段作为上下文信息注入到模型的输入提示（Prompt）中，引导模型基于检索到的事实信息生成回答。

RAG 的典型工作流程分为三个阶段：

（1）**索引阶段（Indexing）**：将原始文档进行预处理，包括文本提取、分块（Chunking）、向量化（Embedding）等操作，构建可供检索的知识库索引。

（2）**检索阶段（Retrieval）**：接收用户查询，将查询文本同样进行向量化处理，通过向量相似度搜索从知识库中找出与查询最相关的文档片段。

（3）**生成阶段（Generation）**：将检索到的文档片段与用户查询一同组织成提示（Prompt），送入大语言模型生成最终回答。

相比纯生成式方法，RAG 具有以下优势：一是有效缓解大模型幻觉问题，生成的回答有据可查；二是知识可动态更新，只需更新外部知识库而无需重新训练模型；三是生成结果可追溯，能够提供回答所依据的原文来源。

### 2.1.2 文档切分策略

文档切分（Chunking）是 RAG 系统中的关键预处理步骤，切分质量直接影响检索效果。切分粒度过大会导致检索结果中包含过多无关信息，稀释关键内容；切分粒度过小则可能破坏语义完整性，导致上下文信息丢失。

常见的切分策略包括固定窗口切分、基于段落的切分和基于语义的切分。本系统采用智能段落切分策略，以自然段落为基本单位，设定合理的字数范围（300-800字），对过短的段落进行合并，对过长的段落按句子边界进行拆分，在保持语义完整性的前提下控制切分粒度。

### 2.1.3 向量检索

向量检索（Vector Search）是基于文本语义进行相似度匹配的检索方法。其基本原理是利用预训练语言模型将文本映射为高维稠密向量（Embedding），语义相似的文本在向量空间中距离较近。检索时，将用户查询同样编码为向量，通过计算查询向量与文档向量之间的相似度（如内积或余弦相似度）找出最相关的文档片段。

FAISS（Facebook AI Similarity Search）是 Meta 公司开源的高效向量相似度搜索库，支持多种索引结构和相似度度量方式，能够在大规模向量集合上实现毫秒级的近似最近邻搜索。本系统使用 FAISS 的 IndexFlatIP（基于内积的精确搜索）作为向量索引，配合 L2 归一化实现余弦相似度检索。

### 2.1.4 BM25 稀疏检索

BM25（Best Matching 25）是经典的基于词频统计的信息检索算法，属于稀疏检索方法。其核心思想是根据查询中各关键词在文档中的出现频率（TF）和逆文档频率（IDF）计算文档与查询的相关性评分。

BM25 的评分公式为：

$$score(D,Q) = \sum_{i=1}^{n} IDF(q_i) \cdot \frac{f(q_i,D) \cdot (k_1+1)}{f(q_i,D) + k_1 \cdot (1-b+b \cdot \frac{|D|}{avgdl})}$$

其中 $f(q_i,D)$ 为词频，$|D|$ 为文档长度，$avgdl$ 为平均文档长度，$k_1$ 和 $b$ 为调节参数。

BM25 在精确关键词匹配场景中表现优异，尤其对专业术语（如航空领域的"起落架""ARINC629 总线"等）具有很高的匹配精度，但在语义理解方面不如稠密向量检索。

### 2.1.5 混合检索与 RRF 融合

混合检索（Hybrid Search）同时利用稠密检索（语义匹配）和稀疏检索（关键词匹配）的互补优势。稠密检索擅长理解查询意图和语义相似性，而稀疏检索擅长精确匹配专业术语和关键词。

倒数排名融合（Reciprocal Rank Fusion, RRF）是一种简单有效的检索结果融合算法，其公式为：

$$RRFscore(d) = \sum_{r \in R} \frac{1}{k + r(d)}$$

其中 $R$ 为所有检索结果列表，$r(d)$ 为文档 $d$ 在列表 $r$ 中的排名，$k$ 为平滑常数（通常取 60）。RRF 不依赖各检索器的具体评分，仅利用排名信息进行融合，具有鲁棒性强、无需调参等优点。

### 2.1.6 交叉编码器重排序

交叉编码器（Cross-Encoder）重排序是一种精细的相关性评估方法。与双编码器（Bi-Encoder）分别对查询和文档独立编码不同，交叉编码器将查询和文档拼接后一起输入到模型中进行联合编码，能够捕获更细粒度的交互信息，从而给出更精确的相关性评分。

由于交叉编码器的计算复杂度较高（需要对每个查询-文档对进行推理），通常采用"先粗筛后精排"的两阶段检索策略：第一阶段使用高效的向量检索或混合检索获取候选文档集，第二阶段使用交叉编码器对候选文档进行重新排序，在保证效率的前提下提升检索精度。

## 2.2 大语言模型（LLM）

### 2.2.1 大语言模型概述

大语言模型（Large Language Model, LLM）是基于 Transformer 架构、在大规模文本语料上训练的深度学习模型，具备强大的自然语言理解和生成能力。以 GPT 系列、Qwen 系列等为代表的大语言模型，通过预训练和指令微调，能够执行问答、摘要、翻译、推理等多种自然语言处理任务。

本系统采用阿里云通义千问（Qwen-plus）模型作为核心推理引擎。Qwen-plus 是阿里巴巴推出的大规模语言模型，在中文理解和生成方面表现优异，特别适合中文专业领域的问答任务。系统通过 DashScope 平台提供的 OpenAI 兼容 API 接口调用模型服务，支持流式输出，即模型可以逐 token 返回生成结果，无需等待完整回答生成完毕。

### 2.2.2 流式生成

流式生成（Streaming Generation）是指大语言模型在推理过程中，生成一个 token 后立即返回给调用方，而不是等待所有 token 生成完毕后一次性返回。这种方式的优势在于：

（1）**降低首次响应延迟**：用户无需等待完整回答，模型生成第一个 token 后即可开始后续处理（如语音合成）。

（2）**支持并行处理**：生成的文本片段可以实时传递给 TTS 模块进行语音合成，实现"边生成边播报"的效果。

（3）**支持打断机制**：在流式生成过程中可以随时中止，实现用户打断功能。

### 2.2.3 提示工程

提示工程（Prompt Engineering）是指通过精心设计输入提示来引导大语言模型生成符合特定要求的输出。在 RAG 系统中，提示通常包括系统角色定义、检索到的参考文档和用户问题三部分。

本系统的提示设计针对语音输出场景进行了专门优化：要求模型生成纯文本回答（不含 Markdown 格式符号），控制回答长度（3-5句话，不超过150字），并在涉及安全关键操作时添加安全提示。

## 2.3 语音识别（STT）

语音识别（Speech-to-Text, STT）是将人类语音信号转换为文本的技术。现代语音识别系统通常采用基于深度学习的端到端模型，直接从原始音频波形或频谱特征映射到文本序列。

本系统采用阿里云智能语音交互（NLS）服务的实时语音识别 API。该服务支持流式音频输入，即客户端可以连续发送音频数据流，服务端实时返回中间识别结果和最终识别结果。主要特性包括：

（1）**实时流式识别**：支持 16kHz 采样率的 PCM 音频流输入，边说边识别，延迟低至毫秒级。

（2）**中间结果回调**：在用户说话过程中持续返回中间识别结果，用于前端实时展示。

（3）**智能断句**：自动检测语句边界，支持配置静音超时时间，在用户停顿时自动断句。

（4）**标点恢复与逆文本归一化**：自动为识别文本添加标点符号，并将口语化数字表达转换为规范格式。

## 2.4 语音合成（TTS）

语音合成（Text-to-Speech, TTS）是将文本转换为语音信号的技术。现代 TTS 系统基于深度学习模型，能够合成接近人类自然语音的高质量语音。

本系统采用阿里云 DashScope 平台提供的 CosyVoice 语音合成服务。CosyVoice 是基于监督语义 token 的多语言零样本语音合成模型，具有以下特点：

（1）**流式合成**：支持双向流式交互，即可以边接收文本边输出音频，无需等待完整文本输入。这一特性使得 TTS 能够与 LLM 的流式生成无缝衔接。

（2）**高质量语音**：生成自然流畅的中文语音，支持多种音色选择。

（3）**低延迟**：从接收到文本到输出第一帧音频的延迟可控制在 200-500 毫秒。

## 2.5 WebSocket 通信协议

WebSocket 是一种在单个 TCP 连接上进行全双工通信的协议。与传统的 HTTP 请求-响应模式不同，WebSocket 在建立连接后，客户端和服务器可以在任意时刻向对方发送数据，无需等待对方的请求。

WebSocket 的全双工特性使其特别适合实时语音交互场景：

（1）**上行**：前端持续向后端发送音频数据流。

（2）**下行**：后端实时向前端推送识别文本、LLM 生成文本和合成音频。

（3）**双向并发**：上行和下行可以同时进行，实现真正的实时交互。

本系统基于 FastAPI 框架的 WebSocket 支持，实现了前后端之间的实时双向通信。

## 2.6 异步编程与并发模型

### 2.6.1 Python asyncio 协程

Python 的 asyncio 是标准库中的异步 I/O 框架，基于事件循环（Event Loop）和协程（Coroutine）实现非阻塞并发。协程通过 async/await 语法，能够在等待 I/O 操作（如网络请求、WebSocket 消息）时让出控制权，由事件循环调度其他就绪的协程执行，从而在单线程内实现高并发。

asyncio 特别适合处理大量 I/O 密集型任务，如 WebSocket 消息收发、LLM API 调用等。本系统的主事件循环运行在单个线程中，所有的 WebSocket 处理、LLM 流式调用和 Pipeline 编排均以协程方式运行。

### 2.6.2 多线程与跨线程通信

对于阻塞型 I/O 操作（如阿里云 NLS SDK 的语音识别启动和停止操作），不能直接在 asyncio 事件循环中执行，否则会阻塞整个事件循环。本系统采用 daemon 线程来运行这些阻塞操作，并通过以下机制实现跨线程安全通信：

（1）**call_soon_threadsafe**：从工作线程向事件循环线程安全地投递回调函数。

（2）**run_coroutine_threadsafe**：从工作线程向事件循环提交协程并获取 Future 对象。

这种协程与线程的混合调度模型，既利用了协程的高效并发能力，又解决了第三方 SDK 的阻塞问题。

## 2.7 前端 Web Audio API

Web Audio API 是浏览器提供的高级音频处理接口，支持音频的采集、处理和播放。本系统利用 Web Audio API 实现以下功能：

（1）**AudioWorklet**：在独立的音频处理线程中运行自定义音频处理逻辑，不阻塞主线程。用于将麦克风采集的音频重采样至 16kHz 并编码为 PCM 格式。

（2）**AudioContext**：管理音频图（Audio Graph），连接音频源、处理节点和输出节点。

（3）**AnalyserNode**：用于实时分析音频信号的频域和时域特征，支持音量可视化和语音活动检测（VAD）。

## 2.8 本章小结

本章介绍了系统涉及的核心技术，包括 RAG 检索增强生成框架、大语言模型及其流式生成技术、语音识别与语音合成的云端 API 服务、WebSocket 实时通信协议、异步编程模型以及前端 Web Audio API。这些技术的有机结合为构建高效、准确的语音问答系统奠定了技术基础。
