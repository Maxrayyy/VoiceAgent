# 第三章 系统需求分析与设计

## 3.1 需求分析

### 3.1.1 功能需求

本系统面向航空维修技术人员，旨在提供基于语音交互的维修知识问答服务。通过对用户需求的深入调研和分析，确定以下功能需求：

（1）**语音输入与识别**：系统应支持用户通过语音方式提问，能够实时采集麦克风音频并将其转换为文本。支持两种录音模式——持续监听模式（系统自动检测用户发言并识别）和按住说话模式（Push-to-Talk, PTT）。识别过程应支持实时展示中间结果，使用户能够看到识别进度。

（2）**知识库检索**：系统应能够从预构建的航空维修知识库中检索与用户问题最相关的文档片段。知识库内容来源于飞机维修手册等权威技术文档，检索结果需包含来源信息（章节、页码等），以便用户追溯和验证。

（3）**智能问答生成**：系统应基于检索到的文档内容，利用大语言模型生成准确、简洁的回答。回答内容需以检索到的事实为依据，对于超出知识库范围的问题应如实告知，避免生成虚假信息。

（4）**语音播报输出**：系统应将生成的文本回答转换为语音进行播报，支持流式合成以降低等待时间。

（5）**多轮对话**：系统应支持多轮对话上下文管理，能够正确理解对话中的代词指代和省略表达，保持对话的连贯性。

（6）**语音打断**：在系统播报回答的过程中，用户应能够随时通过说话或点击按钮打断当前播报，系统立即停止并准备接收新的提问。

（7）**文本查询**：除语音输入外，系统应支持文本输入方式，方便在不便使用语音的场景下使用。

（8）**来源展示**：对于每次回答，系统应展示 RAG 检索所引用的文档来源信息，包括章节名称、页码和相关性评分。

### 3.1.2 非功能需求

（1）**实时性**：系统端到端响应延迟（从用户说完话到开始听到回答）应控制在 2 秒以内，确保交互的自然流畅。

（2）**准确性**：RAG 检索的命中率（Hit Rate@5）应达到 90% 以上，确保检索到的文档与用户问题高度相关。

（3）**可靠性**：系统应具备良好的错误处理机制，在网络波动或服务异常时能够给出友好的提示信息，不会出现未捕获的异常导致服务崩溃。

（4）**可扩展性**：系统架构应支持方便地更换或升级各模块（如更换 LLM 模型、更新知识库文档等），而不需要对系统进行大规模重构。

（5）**易用性**：前端界面应简洁直观，操作流程清晰，用户无需专业培训即可使用。

### 3.1.3 用例分析

系统的核心用例如图 3-1 所示，主要包括以下使用场景：

**用例 1：语音问答**
- 参与者：维修技术人员
- 前置条件：系统已启动，知识库已加载
- 基本流程：用户按下录音按钮或系统自动监听 → 用户说出问题 → 系统识别语音 → 检索知识库 → 生成回答 → 语音播报回答
- 扩展流程：用户打断回答 → 系统停止播报 → 等待新问题

**用例 2：文本问答**
- 参与者：维修技术人员
- 基本流程：用户在输入框输入文本问题 → 系统检索知识库 → 生成回答 → 展示文本回答并语音播报

**用例 3：多轮追问**
- 参与者：维修技术人员
- 前置条件：已完成至少一轮问答
- 基本流程：用户基于上一轮回答继续追问（可使用代词指代）→ 系统结合对话历史改写查询 → 检索并生成回答

**用例 4：知识库管理**
- 参与者：系统管理员
- 基本流程：管理员准备维修文档（PDF）→ 运行文档提取脚本 → 运行索引构建脚本 → 知识库更新完成

## 3.2 系统总体架构设计

### 3.2.1 系统架构概述

本系统采用 Browser/Server（B/S）架构，前端通过浏览器访问，后端基于 Python FastAPI 框架构建。系统总体架构如图 3-2 所示，分为四个层次：

（1）**表示层（Presentation Layer）**：基于原生 HTML/CSS/JavaScript 实现的 Web 前端，负责音频采集与播放、用户交互、状态展示等功能。

（2）**通信层（Communication Layer）**：基于 WebSocket 协议实现前后端实时双向通信，负责音频流上传、文本消息传递和合成音频下发。

（3）**业务逻辑层（Business Logic Layer）**：包含 STT、Pipeline、LLM、TTS 等核心处理模块，负责语音识别、查询处理、知识检索、回答生成和语音合成。

（4）**数据层（Data Layer）**：包含 RAG 知识库索引（FAISS 向量索引、BM25 稀疏索引）和文档元数据，负责知识的存储和检索。

### 3.2.2 端到端流式架构

系统采用端到端流式架构，各模块之间通过流式数据管道相连，实现"边处理边传递"的并行处理模式。整体数据流如下：

```
用户语音 → [前端音频采集] → WebSocket上行 → [STT实时识别] → 识别文本
    → [查询改写] → [RAG混合检索+重排序] → 检索结果
    → [LLM流式生成] → 文本片段 → [TTS流式合成] → 音频片段
    → WebSocket下行 → [前端音频播放] → 用户听到回答
```

与传统的串行处理模式（等待每个模块完全处理完毕后再传递给下一个模块）相比，流式架构的核心优势在于：

- STT 边接收音频边返回识别结果，无需等待用户说完
- LLM 边推理边返回生成的 token，无需等待完整回答
- TTS 边接收文本边合成音频，无需等待完整文本

这种流水线式的并行处理使得各模块的处理时间高度重叠，从而大幅降低端到端延迟。

### 3.2.3 线程与协程混合调度模型

系统采用协程与线程的混合调度模型，根据各组件的 I/O 特性选择最适合的并发方式：

| 组件 | 运行方式 | 设计理由 |
|------|----------|----------|
| WebSocket 消息处理 | asyncio 协程 | 纯异步 I/O，适合协程高并发 |
| LLM 流式调用 | asyncio 协程 | 基于 aiohttp 的异步 HTTP 客户端 |
| Pipeline 编排 | asyncio 协程 | 协调各异步组件 |
| STT 启动/停止 | daemon 线程 | NLS SDK 阻塞操作，不能在事件循环中运行 |
| TTS 合成回调 | SDK 内部线程 | 第三方 SDK 使用自己的线程进行回调 |

跨线程通信通过 `call_soon_threadsafe` 和 `run_coroutine_threadsafe` 两个线程安全方法实现，确保从工作线程投递到事件循环的操作是安全的。

## 3.3 模块设计

### 3.3.1 STT 语音识别模块

STT 模块负责将用户的语音输入转换为文本。模块设计如下：

**类设计**：
- `NlsTokenManager`：NLS 认证 Token 管理器（单例模式），负责自动获取和刷新 Token。
- `StreamingRecognizer`：流式语音识别器，封装阿里云 NLS SDK 的实时语音识别功能。

**接口设计**：
- `start(on_partial, on_final, on_error)`：启动识别会话，注册回调函数
- `feed_audio(audio_data: bytes)`：持续喂入 PCM 音频数据
- `stop() -> str`：停止识别，返回最终识别文本

**关键设计决策**：
- STT 的 start/stop 操作在独立的 daemon 线程中执行，因为 NLS SDK 的连接建立和关闭是阻塞操作
- 通过线程锁和交付标记防止最终结果的重复提交（解决 SDK 回调与 stop 方法之间的竞争条件）

### 3.3.2 RAG 检索模块

RAG 模块负责从知识库中检索与查询相关的文档片段。模块设计采用分层架构：

**类设计**：
- `DocumentStore`：文档存储与检索核心类，整合稠密检索、稀疏检索和重排序功能
- `EmbeddingClient`：文本向量化客户端，调用 DashScope Embedding API
- `BM25Index`：BM25 稀疏检索索引，使用 jieba 分词
- `Reranker`：交叉编码器重排序器，调用 DashScope Rerank API
- `QueryRewriter`：查询改写器，使用 LLM 进行口语化规范化和指代消解

**接口设计**：
- `DocumentStore.search(query, top_k, mode, rerank) -> list`：执行检索，支持 dense/sparse/hybrid 三种模式和可选重排序
- `QueryRewriter.rewrite(query, history) -> str`：改写查询
- `EmbeddingClient.embed_documents(texts) -> ndarray`：批量向量化

**检索流程设计**：
1. 查询改写：结合对话历史进行口语化规范化和指代消解
2. 混合检索：并行执行稠密检索和稀疏检索
3. RRF 融合：使用倒数排名融合算法合并两路检索结果
4. 重排序：使用交叉编码器对融合结果进行精排
5. 返回 Top-K 结果及其元数据

### 3.3.3 LLM 推理模块

LLM 模块负责基于检索结果生成回答文本。

**类设计**：
- `StreamingGenerator`：流式文本生成器，封装 OpenAI 兼容的 DashScope API

**接口设计**：
- `generate(query, context, history) -> AsyncGenerator[str]`：异步生成器，逐步 yield 文本片段

**提示设计**：
- System Prompt 包含角色定义（飞机维修技术顾问）、回答格式约束（纯文本、简洁）和安全提示要求
- 检索到的文档片段以编号列表形式注入到 System Prompt 中
- 对话历史以 user/assistant 消息序列形式传入

### 3.3.4 TTS 语音合成模块

TTS 模块负责将生成的文本转换为语音。

**类设计**：
- `StreamingSynthesizer`：流式语音合成器，封装 DashScope CosyVoice SDK
- `_TtsCallback`：TTS 回调处理类，负责跨线程传递合成音频数据

**接口设计**：
- `start(callback)`：启动合成会话
- `feed_text(text)`：喂入文本片段
- `finish()`：通知文本输入完毕，等待合成完成
- `cancel()`：取消合成（用于打断）

**关键设计决策**：
- 对航空型号中的数字进行预处理（如 B737 → B 七 三 七），避免 TTS 将型号数字读成基数
- TTS 回调在 SDK 内部线程中执行，通过 `call_soon_threadsafe` 将音频数据安全投递到主事件循环

### 3.3.5 Pipeline 编排模块

Pipeline 模块负责协调 RAG、LLM、TTS 三个模块的工作流程。

**类设计**：
- `VoiceChatPipeline`：语音聊天流水线，编排完整的查询处理流程

**接口设计**：
- `process_query(text, on_llm_chunk, on_tts_audio) -> dict`：处理查询，通过回调返回中间结果

**文本缓冲策略**：
- LLM 生成的 token 先进入缓冲区
- 当缓冲区文本遇到句号、问号等标点或累积超过 15 个字时，批量发送给 TTS
- 避免逐 token 发送导致 TTS 合成质量下降

### 3.3.6 WebSocket 服务模块

WebSocket 服务模块负责管理前后端之间的实时通信。

**消息协议设计**：

上行消息（前端 → 后端）：

| 消息类型 | 用途 | 关键字段 |
|---------|------|----------|
| start_recording | 开始录音 | session_id |
| audio | 音频数据 | data (base64), session_id |
| stop_recording | 停止录音 | session_id |
| text_query | 文本查询 | text |
| interrupt | 打断回答 | - |
| clear_history | 清除对话历史 | - |

下行消息（后端 → 前端）：

| 消息类型 | 用途 | 关键字段 |
|---------|------|----------|
| recording_started | 确认开始录音 | - |
| stt_partial | STT 中间结果 | text |
| stt_final | STT 最终结果 | text |
| llm_chunk | LLM 生成文本片段 | text |
| llm_done | LLM 生成完成 | full_text |
| tts_audio | TTS 合成音频 | data (base64) |
| tts_done | TTS 合成完成 | - |
| rag_sources | RAG 检索来源 | sources[] |
| error | 错误信息 | message |

**音频缓冲设计**：
- `AudioBuffer` 类负责将 TTS 回调产生的小音频片段合并为较大的块（8KB）后再通过 WebSocket 发送
- 减少 WebSocket 消息数量，降低网络开销
- 支持跨线程安全投递（TTS 回调线程 → 主事件循环线程）

### 3.3.7 前端模块

前端模块负责音频采集与播放、用户交互和状态展示。

**状态机设计**：
前端维护一个四状态有限状态机：

```
IDLE（空闲）→ LISTENING（录音中）→ PROCESSING（处理中）→ SPEAKING（播报中）→ IDLE
```

各状态下的行为约束：
- IDLE：可开始录音或输入文本
- LISTENING：可停止录音或被自动断句停止
- PROCESSING：等待后端处理，不可操作
- SPEAKING：可打断，可检测到用户语音自动打断

**音频处理设计**：
- 使用 AudioWorklet API（现代浏览器）或 ScriptProcessor API（降级方案）采集音频
- 音频采样率重采样至 16kHz，编码为 PCM 格式，base64 编码后通过 WebSocket 发送
- TTS 音频接收后进入预缓冲队列，累积 1.5 秒后开始播放，避免播放断续

**VAD 语音打断设计**：
- 在 SPEAKING 状态下持续监测麦克风音量
- 使用指数平滑计算背景噪声基线
- 当检测到音量超过基线 3 倍（最低阈值 5.0）且连续 3 帧时，判定用户正在说话
- 触发打断后进入 2 秒冷却期，防止误触发

## 3.4 数据设计

### 3.4.1 文档数据结构

知识库中的每个文档片段（chunk）包含以下数据字段：

```json
{
  "content": "文档文本内容",
  "enriched_content": "上下文增强后的文本内容（可选）",
  "source": "来源文件路径",
  "metadata": {
    "chapter": "章节名称",
    "section": "小节编号",
    "page": 页码数字,
    "chunk_index": 分块序号
  }
}
```

### 3.4.2 索引数据结构

系统维护两套持久化索引：

（1）**FAISS 向量索引**：存储所有文档片段的向量表示，使用 IndexFlatIP（内积精确搜索），配合 L2 归一化实现余弦相似度检索。保存为 `index.faiss` 文件。

（2）**BM25 稀疏索引**：存储 jieba 分词后的文档语料库和 BM25 统计信息。保存为 `bm25_index.pkl`（BM25 模型）和 `bm25_corpus.json`（分词结果）、`bm25_documents.json`（文档数据）三个文件。

（3）**文档元数据**：所有文档片段的完整信息，包括文本内容、来源和元数据。保存为 `documents.json` 文件。

### 3.4.3 评估数据结构

评估数据集包含测试查询及其标准答案，结构如下：

```json
{
  "query": "测试问题文本",
  "golden_content": "标准答案（文档中的精确子串）",
  "source": "来源文件路径"
}
```

## 3.5 本章小结

本章从功能需求和非功能需求两方面对系统进行了全面的需求分析，设计了基于 B/S 架构的四层系统架构，详细描述了各功能模块的类设计、接口设计和关键设计决策，并定义了 WebSocket 消息协议、数据结构和索引方案。系统设计充分考虑了实时性、准确性和可扩展性等非功能需求，为后续的系统实现奠定了良好的基础。
