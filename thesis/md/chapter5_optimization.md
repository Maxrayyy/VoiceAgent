# 第五章 系统优化

本章阐述在 RAG 检索精度和流式交互体验两个维度开展的优化工作。RAG 检索优化聚焦于提升知识检索的准确性与排序质量，流式交互优化聚焦于降低端到端延迟、提升用户体验。

## 5.1 RAG 检索优化

### 5.1.1 混合检索策略

**问题描述**：单一稠密向量检索在航空专业术语（如 "ARINC629 总线"、"TTL 超控"）上易被语义相近但不精确的段落干扰，而 BM25 能实现关键词的精确匹配。

**方案/实现**：在 FAISS 稠密检索基础上，使用 jieba 分词构建 BM25Okapi 索引，并通过倒数排名融合（RRF）合并两路结果：对每个文档按排名位置累加 $1/(k+rank+1)$（k=60），按融合分数降序取 Top-K。

**效果**：相较 baseline，Hit Rate@5 由 0.8667 提升至 0.9333，MRR@5 由 0.7293 提升至 0.8259。

### 5.1.2 交叉编码器重排序

**问题描述**：RRF 仅基于排名融合，无法感知查询-文档之间的细粒度语义关系。

**方案/实现**：采用 "先粗筛后精排" 两阶段策略。第一阶段混合检索返回 4 倍候选文档，第二阶段调用 DashScope `gte-rerank` 交叉编码器对查询-文档对进行联合打分，取 Top-K。

**效果**：Hit Rate@5 由 0.9333 提升至 0.9778，MRR@5 由 0.8259 提升至 0.9667，正确答案被显著推至首位。

### 5.1.3 查询改写

**问题描述**：语音输入的口语化表达（如 "那个飞机轮子下面的东西怎么修"）与维修手册的专业术语差异较大；多轮对话中的代词指代（如 "它怎么修"）若不消解将导致检索失败。

**方案/实现**：使用轻量级模型 Qwen-turbo 结合对话历史改写查询。为控制开销，对长度超过 15 字且无历史的查询启用短路优化直接返回原文；LLM 调用失败时降级为原始查询。

**效果**：口语化问句能被正确映射到专业术语；多轮对话的指代能结合历史被解析，改写后检索命中率明显上升。

### 5.1.4 上下文增强切分

**问题描述**：文档切分后单个 chunk 可能丢失上下文，例如 "检查周期为每 3000 飞行小时" 若缺失 "起落架" 这一主语，与 "起落架检查周期" 查询的匹配度将下降。

**方案/实现**：在索引构建阶段，使用 LLM 分析相邻 chunk 首尾各 200 字，为当前 chunk 生成 30-80 字的语境前缀，拼接到 `enriched_content` 字段后用于向量化。

**效果**：向量表示融入更丰富的语境信息，稠密检索的语义匹配准确性得到提升。

### 5.1.5 航空术语自定义词典

**问题描述**：jieba 默认词典对航空术语识别有限，"起落架" 可能被切分为 "起落"+"架"，"液压系统" 被切分为 "液压"+"系统"，BM25 匹配因此失准。

**方案/实现**：整理航空维修领域术语词典 `data/aviation_terms.txt`，在构建 BM25 索引前通过 `jieba.load_userdict()` 加载，确保专业术语整体分词。

**效果**：BM25 对专业术语的召回显著改善，与稠密检索形成更有效的互补。

### 5.1.6 元数据过滤

**问题描述**：部分查询带有明确的章节或页码约束，纯语义检索无法精确利用这些结构化信息。

**方案/实现**：文档切分时自动提取章节标题、小节编号与页码作为 metadata，检索接口支持按 `chapter`、`page_range` 等字段匹配，若 metadata 不满足任一过滤条件则过滤掉该文档。

**效果**：对带章节/页码约束的查询检索精度提升，并能在大规模知识库下大幅缩小候选集合。

## 5.2 流式交互优化

### 5.2.1 端到端流式并行处理

**问题描述**：传统串行模式下端到端延迟 ≈ STT 200 ms + RAG 400 ms + LLM 完整生成 3000 ms + TTS 完整合成 2000 ms ≈ 5600 ms。

**方案/实现**：STT 流式识别、LLM 流式生成与 TTS 流式合成三阶段并行推进。LLM 一旦输出首句即触发 TTS 合成，TTS 一旦产生首帧音频即下发到前端，LLM 与 TTS 耗时高度重叠。

**效果**：端到端延迟 ≈ STT 200 ms + RAG 400 ms + LLM 首 token 500 ms + 文本缓冲 200 ms + TTS 首帧 300 ms ≈ 1600 ms，较串行模式降低约 71%。

### 5.2.2 文本缓冲策略优化

**问题描述**：若将 LLM 输出的每个 token 直接送入 TTS 会因片段过短影响合成韵律；若等全部文本又丧失流式优势。

**方案/实现**：在 LLM 与 TTS 之间引入文本缓冲层，采用双触发策略：缓冲区累计超过 15 字时刷新；或遇到句号、问号、感叹号、分号、逗号等标点时立即刷新。

**效果**：TTS 每次收到语义完整的短句，合成自然度优于逐 token 输入，同时首帧延迟维持在 200 ms 左右。

### 5.2.3 音频预缓冲策略

**问题描述**：若收到首帧音频立即播放，容易因后续音频尚未到达或网络抖动出现播放断续。

**方案/实现**：前端使用 Web Audio API 的 AudioContext 调度机制，累计约 1.5 秒音频后再开始播放，并按精确时间戳对每段音频进行调度。

**效果**：消除首段卡顿，平滑网络波动，相邻音频片段无缝衔接。

### 5.2.4 AudioBuffer 批量发送

**问题描述**：TTS SDK 每次回调的音频仅几百字节，若逐次经 WebSocket 发送会产生大量小消息，增加网络与前端处理开销。

**方案/实现**：在后端引入 AudioBuffer 将 SDK 回调的小片段合并到 8 KB 的块再统一下发；该缓冲层也承担将音频从 SDK 回调线程安全投递到 asyncio 事件循环线程的职责。

**效果**：WebSocket 消息数量显著下降，前端解码开销降低，同时保证线程安全。

### 5.2.5 语音打断机制

**问题描述**：系统播报过程中若用户想补充或重新提问，需要即时终止当前回答。

**方案/实现**：① 手动打断，前端点击按钮发送 `interrupt` 消息；② VAD 语音打断，SPEAKING 状态下前端以指数平滑跟踪背景噪声基线，当音量连续 3 帧超过基线 3 倍即触发。打断后后端递增 `query_generation` 计数使 LLM 生成循环 break、调用 TTS cancel 并清空 AudioBuffer，前端随即停止播放并回到 LISTENING 状态。

**效果**：打断响应时间控制在百毫秒级，交互自然度显著提升。

### 5.2.6 STT 静音容忍优化

**问题描述**：NLS SDK 默认的 `max_sentence_silence` 约 800 ms，用户思考停顿就可能被判定为句末，一句话被切成多段。

**方案/实现**：将参数放宽至 1500 ms，并通过环境变量 `STT_MAX_SILENCE` 支持灵活配置。

**效果**：句内停顿不再误切，多轮对话的语义完整性明显改善。

## 5.3 本章小结

本章从 RAG 检索与流式交互两个维度阐述了系统的优化工作。RAG 方面，通过混合检索、交叉编码器重排序、查询改写、上下文增强切分、航空术语词典和元数据过滤六项措施，Hit Rate@5 由 0.8444 提升至 0.9778，MRR@5 由 0.6626 提升至 0.9667。流式交互方面，通过端到端并行处理、文本缓冲、音频预缓冲、批量发送、多层级打断和 STT 静音容忍等优化，端到端延迟由约 5.6 秒降至约 1.6 秒，降低约 71%，同时保持了良好的合成自然度与交互流畅性。
