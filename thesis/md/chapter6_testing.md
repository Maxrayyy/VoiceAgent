## 6.4 RAG 检索性能评估

### 6.4.1 评估方法

在 45 条查询上采用三项标准指标（K=5）：

① Hit Rate@K：$HitRate@K = \frac{1}{|Q|} \sum_{q \in Q} \mathbb{1}[\exists d \in TopK(q): d \text{ is relevant}]$，衡量基本召回能力。
② MRR@K：$MRR@K = \frac{1}{|Q|} \sum_{q \in Q} \frac{1}{rank_q}$，衡量正确文档排名位置。
③ nDCG@K：$nDCG@K = DCG@K / IDCG@K$，$DCG@K = \sum_{i=1}^{K} rel_i / \log_2(i+1)$，综合衡量排序质量。

### 6.4.2 对比实验设计

设计四组实验：实验 1 纯稠密检索（baseline）；实验 2 混合检索（dense + BM25 + RRF）；实验 3 混合检索 + 交叉编码器重排序；实验 4 在实验 3 之上叠加查询改写、上下文增强切分与术语词典的完整优化版。所有实验 Top-K=5。

### 6.4.3 实验结果

四组实验的指标对比如表 6.4 所示。

表 6.4 不同检索策略的性能对比

| 检索策略 | Hit Rate@5 | MRR@5 | nDCG@5 |
|---------|-----------|-------|--------|
| baseline（dense） | 0.8444 | 0.6626 | 0.7071 |
| baseline（dense，优化 Embedding） | 0.8667 | 0.7293 | 0.7601 |
| hybrid | 0.9333 | 0.8259 | 0.8537 |
| hybrid + rerank | 0.9778 | 0.9667 | 0.9678 |

### 6.4.4 结果分析

（1）引入混合检索后 Hit Rate +7.7%、MRR +13.2%、nDCG +12.3%，BM25 对 "ARINC629" 等专业术语的精确匹配与稠密检索的语义理解形成互补。
（2）引入交叉编码器重排序后 Hit Rate +4.8%、MRR +17.1%、nDCG +13.4%，MRR 大幅提升表明重排序显著将正确答案推至首位。
（3）最终系统 Hit Rate@5 = 97.78%（45 条中 44 条命中）、MRR@5 = 0.9667、nDCG@5 = 0.9678，累计较 baseline 分别提升 12.5%、30.3%、25.7%。

## 6.5 延迟性能分析

### 6.5.1 各模块延迟分解

通过 e2e 客户端采集的各模块典型延迟如表 6.5 所示。

表 6.5 各模块延迟分解

| 处理阶段 | 典型延迟 | 说明 |
|---------|---------|------|
| STT 实时识别 | ~200 ms | 边说边识别 |
| 查询改写 | 100-300 ms | Qwen-turbo 单次调用 |
| RAG 混合检索 | 200-500 ms | FAISS + BM25 + RRF |
| 重排序 | 100-300 ms | gte-rerank API |
| LLM 首 token | 300-800 ms | Qwen-plus 流式推理启动 |
| TTS 首帧音频 | 200-500 ms | CosyVoice 流式合成启动 |

### 6.5.2 端到端延迟测量方法与结果

采用 `scripts/e2e_test_client.py` 作为测量工具，以 WebSocket 客户端向后端重放预合成的测试音频，客户端在收到 `stop_recording` 确认、`stt_final`、`llm_chunk` 首帧、`tts_audio` 首包四个关键事件处记录时间戳，核心指标取：

① 识别收尾：`stop_recording` → `stt_final`；
② 检索+首 token：`stt_final` → `llm_chunk` 首帧（含查询改写、混合检索、重排序、LLM 首 token）；
③ TTS 首帧：首个 `llm_chunk` → 首包 `tts_audio`（含文本缓冲与 CosyVoice 首包时延）；
④ 端到端首响：`stop_recording` → 首包 `tts_audio`，作为"说完到听到"指标。

在典型测试集上，串行模式（待 LLM 完整生成后一次性合成）端到端延迟约 5.6 s；流式模式下 LLM 与 TTS 高度并行，"说完到首帧音频"典型值约 1.6 s，较串行模式降低约 71%。

## 6.6 本章小结

本章通过单元测试、端到端测试、RAG 检索性能评估与延迟分析验证了系统的正确性与性能。16 个单元测试全部通过；端到端测试覆盖主要交互路径；四组对比实验定量验证了各优化措施的贡献，最终 Hit Rate@5 达 97.78%、MRR@5 达 0.9667、nDCG@5 达 0.9678；流式架构将端到端延迟从约 5.6 秒降至约 1.6 秒。
