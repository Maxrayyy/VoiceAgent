# 第二章 技术选型

本章介绍系统核心技术及其方案对比与选型理由。

## 2.1 检索增强生成（RAG）

### 2.1.1 RAG 基本原理

检索增强生成（Retrieval-Augmented Generation, RAG）[1] 将信息检索与文本生成结合，用于缓解 LLM 的幻觉与知识时效问题。其流程分三阶段：（1）索引：对文档提取、分块与向量化建索引；（2）检索：将查询向量化并召回 Top-K 片段；（3）生成：将片段与问题拼为提示输入 LLM 生成回答。RAG 具有缓解幻觉、知识热更新、可溯源三大优势，适合航空维修这类准确性要求高的场景。

### 2.1.2 文档切分策略

在 RAG 系统中，文档切分是知识库构建阶段的基础环节，其作用是将原始长文档拆分为适合索引、检索与生成使用的片段。切分策略是否合理，会直接影响向量表示质量、召回结果相关性以及后续大语言模型所接收上下文的可用性。如果片段长度过长，单个片段中容易混入过多无关信息，使查询与目标内容之间的对应关系被弱化；如果片段长度过短，又可能打散原本完整的操作条件、对象指代和结论说明，从而削弱语义完整性。因此，文档切分并不是简单的文本分块，而是需要在检索效率与语义保真之间取得平衡。常见方案对比如表 2.1 所示。

表 2.1 文档切分方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| 固定窗口 | 简单、均匀 | 易截断句子 | 结构弱文档 |
| 段落切分 | 保留语义边界 | 段长差异大 | 结构化手册 |
| 语义切分 | 语义完整性最优 | 依赖模型、开销大 | 类型多样文档 |

从表 2.1 可以看出，固定窗口切分实现简单，但容易在句子中间或条件说明尚未结束时发生截断；语义切分虽然理论上能够更好保持内容完整，但其效果较依赖额外模型，且处理成本相对较高。结合航空维修手册章节结构清晰、段落边界稳定、技术要点往往以自然段为单位组织的特点，本文最终选用**段落切分**策略。在具体处理过程中，系统优先保留原始段落结构，并将片段长度控制在 300—800 字之间：对过短段落进行合并，对过长段落再按句子边界适度拆分。这样既能够较好保留原始语义单元，又能避免片段长度差异过大，从而为后续检索和生成提供更稳定的输入。

### 2.1.3 向量检索

向量检索是当前 RAG 系统中最常见的语义召回方式，其基本思想是利用预训练嵌入模型 [2] 将文本映射为高维稠密向量，并通过向量之间的距离或相似度来衡量语义相关性。与传统关键词匹配不同，向量检索能够在词汇不完全重叠的情况下发现潜在语义关联，例如将“飞机轮子下面的支撑”与“起落架”建立联系。这一能力对于语音问答场景尤为重要，因为用户的口语化表达往往与手册中的规范术语并不完全一致。考虑到向量检索需要依赖专门的向量索引或向量数据库支撑，表 2.2 对几种常见方案进行了比较。

表 2.2 向量库方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| FAISS | 开源、单机性能强 | 无分布式 | 中小规模索引 |
| Milvus | 分布式、海量 | 部署复杂 | 亿级生产环境 |
| Chroma | 轻量、Python 友好 | 大规模性能一般 | 原型开发 |
| Pinecone | 全托管云服务 | 收费、数据出境 | 国外云端 |

从工程实现角度看，向量检索方案的选择既与数据规模有关，也与部署复杂度和实验环境有关。Milvus 等分布式方案更适合海量数据场景，但对于毕业设计阶段的中小规模知识库而言，其部署与维护成本相对较高；Chroma 和 Pinecone 虽然也具有一定易用性，但前者在大规模场景下性能上限有限，后者则涉及云端托管与额外成本问题。综合考虑当前知识库规模处于万级片段量级、实验环境以单机部署为主、并且需要较强的本地可控性，本文最终选用 **FAISS** 作为向量检索基础设施。该方案能够在单机环境下提供较高的检索效率，且配合向量归一化处理后，便于用统一的相似度度量完成排序。

### 2.1.4 BM25 稀疏检索与混合检索

虽然向量检索能够较好处理口语化表达和语义近义问题，但在航空维修场景中，许多查询还包含型号、章节号、部件编号和缩略语等高精度术语信息。对于这类内容，仅依赖语义相似度往往并不稳妥，仍然需要借助关键词匹配能力更强的稀疏检索方法。BM25 [3] 是最具代表性的稀疏检索模型之一，其通过词频（TF）与逆文档频率（IDF）综合计算查询与文档之间的相关性，并利用 $k_1$ 与 $b$ 两个参数调节词频饱和和长度归一化效果。相比稠密检索，BM25 对“ARINC629 总线”“TTL 超控”等专业术语具有更强的精确匹配能力，但也存在语义泛化不足的问题。

正因为两类检索方法各有优势，当前许多 RAG 系统都会采用混合检索策略 [4]，即在保留向量检索语义召回能力的同时，引入 BM25 处理术语和关键词敏感问题，再通过融合方法整合两路结果。常见融合策略如表 2.3 所示。

表 2.3 混合检索融合方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| RRF | 仅依赖排名、无需校准 | 忽略分数差异 | 通用异构融合 |
| 加权融合 | 权重灵活 | 需分数归一化 | 分数分布相近 |
| 学习融合 | 上限高 | 需标注数据 | 有标注业务 |

在这些方法中，加权融合虽然灵活，但需要先解决不同检索器分数空间不一致的问题；学习融合理论上能够取得更高上限，但通常依赖额外标注数据和训练过程。相比之下，RRF 更适合当前这种缺乏大规模人工标注、但需要较快完成系统构建与验证的场景。本文因此选用 **RRF** 作为融合策略，其计算公式为 $RRFscore(d) = \sum_{r \in R} \frac{1}{k + r(d)}$，其中 $k$ 常取 60。该方法主要依据候选结果的排序位置进行融合，而不直接依赖原始分数，从而避免了复杂的分数校准过程，并在异构检索结果整合中表现出较好的稳健性。

### 2.1.5 交叉编码器重排序

在完成初步召回之后，系统还需要面对另一个问题，即“召回到的候选片段中，哪些最值得送入大语言模型”。由于语言模型上下文容量有限，如果前排结果中混入大量主题相近但并非最相关的片段，就会削弱最终回答质量。为此，RAG 系统通常会在召回阶段之后增加重排序环节，对候选集合进行更细粒度的相关性判断。

交叉编码器（Cross-Encoder）[5] 是常用的重排序方法之一。它将查询与文档拼接后进行联合编码，使模型能够直接建模二者之间的词级和句级交互，因此在排序精度上通常优于双编码器方案。但与此同时，这种方法需要对每个候选片段分别进行推理，计算复杂度相对较高，因此更适合用于“少量候选的精排”而非“全量文档的初筛”。基于这一特点，本文采用“粗筛 + 精排”的两阶段策略：先通过混合检索召回约 20 个候选片段，再利用交叉编码器对其进行重排序，并从中选取 Top-3 至 Top-5 结果提供给后续生成模块。

## 2.2 大语言模型（LLM）

### 2.2.1 大语言模型概述

大语言模型（LLM）是本项目实现智能问答能力的核心组件。其基础通常建立在 Transformer 架构 [6] 之上，通过自注意力机制建模长距离依赖，并在大规模语料预训练和后续指令微调基础上获得较强的语言理解与生成能力。对于本项目而言，LLM 的作用并不是脱离知识库自由作答，而是在检索结果约束下完成问题理解、答案组织与自然语言表达。因此，模型选型需要同时考虑中文能力、接口稳定性、成本水平以及与现有工程体系的集成便利性。表 2.4 给出了几种常见方案的对比。

表 2.4 主流 LLM 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| Qwen-Plus | 中文强、API 成熟、成本低 | 海外可用性差 | 国内中文问答 |
| GPT-4 | 综合推理最强 | 贵、数据出境 | 国际化推理 |
| LLaMA-3 | 开源可私有化 | 中文语料少 | GPU 私有部署 |
| DeepSeek | 中文强、开源 | 生态完善中 | 成本敏感 |
| GLM | 国产合规、中文好 | 生态略弱 | 政企合规 |

综合比较可以看出，GPT-4 在综合推理能力上具有优势，但成本较高，且涉及数据出境问题；LLaMA-3 等开源模型虽然便于私有化部署，但在中文专业问答场景下仍需要较多额外适配；GLM 和 DeepSeek 也具有一定可行性，但在当前项目环境下，接入成熟度和整体稳定性仍需结合具体生态考量。基于中文能力、接口成熟度、调用成本和实验可复现性的综合平衡，本文最终选用 **Qwen-Plus** 作为主要生成模型。该模型能够较好满足专业问答场景下的回答质量要求，同时也便于根据任务复杂度进一步切换到成本更低的辅助模型。

### 2.2.2 流式生成

在语音问答场景中，语言模型如果必须等待整段回答全部生成完成后再统一返回，往往会显著增加用户等待时间。因此，本文采用流式生成方式，让模型在自回归解码过程中逐步输出文本片段，而不是一次性返回完整结果。这样做首先能够缩短用户感知到的首段响应时间，因为系统无需等待全部内容完成后才开始后续处理；其次，增量输出的文本可以被及时送入语音合成模块，从而形成“生成、合成、播报”相互重叠的处理链路；再次，流式生成还便于与打断机制结合，使系统能够在用户提出新问题后及时停止旧回答继续向下游传播。对于强调实时交互的语音问答项目而言，流式生成并不是附加优化，而是影响整体体验的关键能力。

### 2.2.3 提示工程

在 RAG 框架下，语言模型最终生成什么样的回答，不仅取决于检索结果本身，还取决于提示如何组织。对于本项目而言，提示工程的目标并不是单纯追求回答“更像聊天”，而是让回答既符合专业场景要求，又适合语音播报。为此，提示内容通常由系统指令、检索文档和用户问题三部分共同构成，其中系统指令负责限定回答边界，检索文档提供事实依据，用户问题则决定本轮生成目标。

考虑到输出结果最终需要进入语音合成环节，本文在提示设计上做了针对性约束。首先，回答中应尽量避免 Markdown 等不适合朗读的格式化标记，以免影响语音播报自然度；其次，回答长度需要保持克制，通常限制在 3—5 句、总长度不超过 150 字，以适配实时播报场景；再次，对于涉及安全关键操作的问题，需要在回答中明确提醒用户以官方维修手册为准；最后，当检索证据不足时，模型应直接说明依据有限，而不是在缺乏支持的情况下继续扩展生成。通过这些约束，提示工程在一定程度上把语言模型从“自由生成”转化为“受证据约束、面向播报输出”的专业问答组件。

## 2.3 语音交互与实时通信

### 2.3.1 语音识别（STT）

语音识别（STT）承担将用户语音输入实时转换为文本的任务，是整条语音问答链路的起点。对于航空维修场景而言，识别模块不仅需要具备较好的中文准确率，还需要支持连续流式输入、中间结果实时返回以及较低延迟，否则后续检索与生成环节将缺乏稳定且及时的文本输入。当前主流 STT 方案大多建立在 Transformer 等端到端模型 [7] 基础之上，但在部署模式、实时性、中文能力和工程接入成本方面仍然存在明显差异。表 2.5 对几种典型方案进行了比较。

表 2.5 STT 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| 阿里云 NLS | 流式低延迟、中文精度高 | 商业 API | 中文实时交互 |
| 百度语音 | 中文生态成熟 | 流式体验略差 | 中文识别 |
| 讯飞语音 | 方言精度高 | 价格较高 | 多方言客服 |
| 本地 Whisper | 开源多语种 | 中文弱、需 GPU | 离线私有 |
| Paraformer | 中文好、推理快 | 需自建链路 | 私有化部署 |

从对比结果来看，本地部署方案虽然在数据可控性方面具有优势，但往往对算力和部署维护提出更高要求；部分商业方案在中文识别方面表现良好，但在实时反馈能力或成本上存在不同程度约束。综合考虑本项目以中文语音问答为主、强调低延迟流式交互、同时希望降低本地算力依赖的实际需求，本文最终选用 **阿里云 NLS** 作为语音识别方案。该方案支持流式识别与中间结果实时反馈，能够较好适应“边说边识别”的交互方式；同时可配置静音判定时间，对句内停顿具有一定容忍度，并具备标点恢复与数字规范化等实用能力。

### 2.3.2 语音合成（TTS）

语音合成（TTS）负责将文本回答转化为用户可直接收听的语音输出，是语音问答系统面向用户呈现结果的最后一个关键环节。与传统播报场景相比，本项目中的 TTS 不仅要保证中文语音自然度，还要尽量缩短首帧返回时间，并支持与语言模型流式输出协同工作。近年来，基于 Tacotron、FastSpeech、VITS 等端到端神经网络方法的 TTS 技术 [8] 在自然度方面已经明显优于传统拼接式合成，但不同方案在流式能力、中文表现和部署成本方面仍有差异。表 2.6 给出了几种可选方案的对比。

表 2.6 TTS 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| CosyVoice | 双向流式、中文自然 | 定制需训练 | 中文实时播报 |
| 阿里 NLS TTS | 接入简单 | 流式较弱 | 一般播报 |
| Azure TTS | 多语种多音色 | 数据出境、延迟高 | 国际化多语种 |
| OpenAI TTS | 英文自然 | 中文一般、海外 | 英文内容 |

从应用需求出发，语音问答项目更看重的是“能否边生成边播报”，而不仅是离线场景下的单次合成质量。相比之下，支持双向流式能力的方案更适合本项目的整体链路设计。综合考虑中文语音自然度、流式合成能力以及工程接入便利性，本文最终选用 **CosyVoice** 作为语音合成方案。该方案能够在接收文本片段的同时持续返回音频数据，因此较适合与流式语言模型生成过程衔接；同时，其中文发音自然度较高，能够满足专业问答场景的播报需求。

### 2.3.3 异步并发模型与 WebSocket 通信

语音问答与普通文本问答的一个重要区别在于，它需要在同一轮交互中同时处理音频上行、识别结果回传、文本回答下发以及语音音频播放等多种数据流。这意味着系统不仅需要较强的实时双向通信能力，还需要能够协调多个处理阶段并发执行，否则端到端时延会快速累积。就通信协议而言，不同方案在连接方式、双向能力和实现复杂度方面存在差异，表 2.7 对常见协议进行了比较。

表 2.7 实时通信协议对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| WebSocket | 单连接全双工、开销低 | 需专门服务端 | 实时双向交互 |
| SSE | 简单、基于 HTTP | 仅单向 | LLM 流式推送 |
| 长轮询 | 兼容性最好 | 延迟高 | 老旧环境 |
| WebRTC | P2P 低延迟 | 协议复杂 | 多端音视频 |

从表 2.7 可以看出，SSE 更适合单向推送，长轮询虽然兼容性较好，但在频繁交互场景中延迟较高，WebRTC 则更适合复杂音视频通信。对于本项目这种需要持续上传音频、持续回传文本与音频结果的实时语音问答场景而言，**WebSocket** [9] 更符合通信需求。该协议在建立连接后能够持续保持双向通信能力，因此既适合上行传输音频数据，也适合下行返回识别文本、检索来源、回答片段和语音音频。

在并发模型方面，本文采用协程与多线程混合调度方式。这样设计的原因在于，浏览器连接管理、消息收发和语言模型调用本质上属于典型的异步 I/O 任务，适合放在统一事件循环中处理；而语音识别与语音合成服务在实际接入中又伴随阻塞式连接建立、关闭和回调触发等特征，如果直接放入主流程，就会影响整个系统的响应能力。通过将异步业务逻辑与阻塞式语音服务调用分开处理，系统既能够保留异步并发在高频消息处理上的优势，又能够兼容第三方语音服务 SDK 的阻塞特性，从而满足实时语音问答对吞吐能力和稳定性的双重要求。

## 2.4 本章小结

本章围绕项目实现所涉及的关键技术与方案选择展开分析。知识检索部分明确了段落切分、FAISS、RRF 融合和交叉编码器重排序等方案的适用性；语言模型部分说明了 Qwen-Plus、流式生成和语音场景提示工程的选择依据；语音交互部分则确定了阿里云 NLS、CosyVoice、WebSocket 以及协程与线程混合调度方式。上述技术选型共同构成了后续系统设计与实现的基础。
