# 第二章 技术选型

本章介绍系统核心技术及其方案对比与选型理由。

## 2.1 检索增强生成（RAG）

### 2.1.1 RAG 基本原理

检索增强生成（Retrieval-Augmented Generation, RAG）[1] 将信息检索与文本生成结合，用于缓解 LLM 的幻觉与知识时效问题。其流程分三阶段：（1）索引：对文档提取、分块与向量化建索引；（2）检索：将查询向量化并召回 Top-K 片段；（3）生成：将片段与问题拼为提示输入 LLM 生成回答。RAG 具有缓解幻觉、知识热更新、可溯源三大优势，适合航空维修这类准确性要求高的场景。

### 2.1.2 文档切分策略

粒度过大稀释信息，过小则破坏语义完整性。对比见表 2.1。

表 2.1 文档切分方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| 固定窗口 | 简单、均匀 | 易截断句子 | 结构弱文档 |
| 段落切分 | 保留语义边界 | 段长差异大 | 结构化手册 |
| 语义切分 | 语义完整性最优 | 依赖模型、开销大 | 类型多样文档 |

本项目选用**段落切分**。维修手册段落结构明显，天然段落即完整技术要点；设 300—800 字窗口，过短合并、过长按句号拆分，兼顾语义完整性与块长一致性。

### 2.1.3 向量检索

向量检索利用预训练嵌入模型 [2] 将文本映射为高维稠密向量，使语义相似度转化为向量距离计算，可召回词汇不重叠但语义相关的文档（如"飞机轮子下面的支撑"→"起落架"）。向量库对比见表 2.2。

表 2.2 向量库方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| FAISS | 开源、单机性能强 | 无分布式 | 中小规模索引 |
| Milvus | 分布式、海量 | 部署复杂 | 亿级生产环境 |
| Chroma | 轻量、Python 友好 | 大规模性能一般 | 原型开发 |
| Pinecone | 全托管云服务 | 收费、数据出境 | 国外云端 |

本项目选用 **FAISS**。当前知识库规模处于万级片段量级，采用基于向量相似度的精确检索方式即可在单机环境下实现毫秒级响应；配合向量归一化处理后，也便于用统一的相似度度量完成排序。

### 2.1.4 BM25 稀疏检索与混合检索

BM25 [3] 是稀疏检索代表，基于词频（TF）与逆文档频率（IDF）计算相关性，通过 $k_1$（词频饱和，常取 1.2—2.0）与 $b$（长度归一化，0.75）调节。BM25 对"ARINC629 总线""TTL 超控"等专业术语精确匹配强但缺乏语义泛化。混合检索 [4] 将两路融合取长补短，策略对比见表 2.3。

表 2.3 混合检索融合方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| RRF | 仅依赖排名、无需校准 | 忽略分数差异 | 通用异构融合 |
| 加权融合 | 权重灵活 | 需分数归一化 | 分数分布相近 |
| 学习融合 | 上限高 | 需标注数据 | 有标注业务 |

本项目选用 **RRF**，公式为 $RRFscore(d) = \sum_{r \in R} \frac{1}{k + r(d)}$，$k$ 常取 60。该方法仅依赖排序位置而不直接依赖原始分数，因此无需复杂的分数校准，在缺少人工标注数据时具有较好的稳健性。

### 2.1.5 交叉编码器重排序

交叉编码器（Cross-Encoder）[5] 将查询与文档拼接联合编码，自注意力建模细粒度交互，精度高于双编码器但复杂度 O(n)。工程上采用"粗筛+精排"：先由混合检索召回约 20 个候选，再由交叉编码器精排取 Top-3~5。

## 2.2 大语言模型（LLM）

### 2.2.1 大语言模型概述

大语言模型（LLM）基于 Transformer 架构 [6]，依赖自注意力建模长距离依赖，经预训练与指令微调获得语言理解与生成能力。对比见表 2.4。

表 2.4 主流 LLM 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| Qwen-Plus | 中文强、API 成熟、成本低 | 海外可用性差 | 国内中文问答 |
| GPT-4 | 综合推理最强 | 贵、数据出境 | 国际化推理 |
| LLaMA-3 | 开源可私有化 | 中文语料少 | GPU 私有部署 |
| DeepSeek | 中文强、开源 | 生态完善中 | 成本敏感 |
| GLM | 国产合规、中文好 | 生态略弱 | 政企合规 |

本项目选用 **Qwen-Plus**。该模型在中文理解与生成方面表现较好，接口接入方式成熟，能够满足专业问答场景下对稳定性与成本控制的双重要求；同时，也便于在不同复杂度任务之间进行模型切换。

### 2.2.2 流式生成

LLM 采用自回归生成，每预测一个 token 即通过 SSE 或 WebSocket 推送。其价值有三：①降低首 token 延迟（300—800 ms）；②支持下游并行，文本片段可实时交 TTS 实现"生成—合成—播报"流水线；③支持打断，可随时中止以响应新输入。

### 2.2.3 提示工程

RAG 提示由系统指令、检索文档与用户问题组成。针对语音输出场景，本项目在提示设计上做了四点优化：（1）避免使用不适合朗读的格式化标记；（2）限制回答为 3—5 句、总长度不超过 150 字；（3）涉及安全关键操作时追加风险提示；（4）资料缺失时明确说明依据不足，避免无据生成。

## 2.3 语音交互与实时通信

### 2.3.1 语音识别（STT）

语音识别（STT）将语音转为文本，主流方案基于 Transformer 端到端模型 [7]，对比见表 2.5。

表 2.5 STT 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| 阿里云 NLS | 流式低延迟、中文精度高 | 商业 API | 中文实时交互 |
| 百度语音 | 中文生态成熟 | 流式体验略差 | 中文识别 |
| 讯飞语音 | 方言精度高 | 价格较高 | 多方言客服 |
| 本地 Whisper | 开源多语种 | 中文弱、需 GPU | 离线私有 |
| Paraformer | 中文好、推理快 | 需自建链路 | 私有化部署 |

本项目选用 **阿里云 NLS**。该方案支持流式识别与中间结果实时反馈，能够较好适应“边说边识别”的实时交互需求；同时可配置静音判定时间，对句内停顿具有一定容忍度，并具备标点恢复与数字规范化等实用能力。

### 2.3.2 语音合成（TTS）

语音合成（TTS）将文本转为语音波形，端到端神经网络方案（Tacotron、FastSpeech、VITS 等）[8] 在自然度上远超拼接合成，对比见表 2.6。

表 2.6 TTS 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| CosyVoice | 双向流式、中文自然 | 定制需训练 | 中文实时播报 |
| 阿里 NLS TTS | 接入简单 | 流式较弱 | 一般播报 |
| Azure TTS | 多语种多音色 | 数据出境、延迟高 | 国际化多语种 |
| OpenAI TTS | 英文自然 | 中文一般、海外 | 英文内容 |

本项目选用 **CosyVoice**。该方案支持双向流式合成，可以在接收文本片段的同时持续返回音频数据，较适合与流式语言模型生成过程衔接；同时，其中文发音自然度较高，能够满足专业问答场景的播报需求。

### 2.3.3 异步并发模型与 WebSocket 通信

语音问答对实时双向通信与高并发要求较高，协议对比见表 2.7。

表 2.7 实时通信协议对比

| 方案 | 优点 | 缺点 | 适用场景 |
|-|-|-|-|
| WebSocket | 单连接全双工、开销低 | 需专门服务端 | 实时双向交互 |
| SSE | 简单、基于 HTTP | 仅单向 | LLM 流式推送 |
| 长轮询 | 兼容性最好 | 延迟高 | 老旧环境 |
| WebRTC | P2P 低延迟 | 协议复杂 | 多端音视频 |

本项目选用 **WebSocket** [9]。该协议在建立连接后能够持续保持双向通信能力，因此既适合上行传输音频数据，也适合下行返回识别文本、检索来源、回答片段和语音音频。与长轮询等方式相比，WebSocket 更适合语音问答中这种消息频繁、交互连续的实时场景。

在并发模型方面，项目采用协程与多线程混合调度方式。其基本思路是，将浏览器连接管理、消息收发和语言模型推理这类异步 I/O 任务放在统一的事件循环中处理，以提高并发效率；而将语音识别与语音合成服务中带有阻塞特征的连接建立、关闭和回调处理放入独立线程中执行，避免其影响主流程的响应能力。这样既能保留异步并发在高频消息处理上的优势，又能兼容第三方语音服务 SDK 的阻塞特性，从而满足实时语音问答对吞吐能力和稳定性的双重要求。

## 2.4 本章小结

本章围绕项目实现所涉及的关键技术与方案选择展开分析。知识检索部分明确了段落切分、FAISS、RRF 融合和交叉编码器重排序等方案的适用性；语言模型部分说明了 Qwen-Plus、流式生成和语音场景提示工程的选择依据；语音交互部分则确定了阿里云 NLS、CosyVoice、WebSocket 以及协程与线程混合调度方式。上述技术选型共同构成了后续系统设计与实现的基础。
