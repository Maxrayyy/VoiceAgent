# 论文详细修改意见

## 1. 目前的整体判断

结合 `thesis/md/` 全文与 `src/` 实现代码来看，当前论文已经具备完整的章节框架、图表基础和实验数据，但仍然存在较明显的“技术说明文”写法痕迹，距离毕业论文要求的“论述性、解释性、学术性表达”还有一段距离。你总结的第 5、6、9、10、11 点判断基本准确，而且这些问题不是局部问题，而是贯穿第二章到第六章的行文方式问题。

当前最需要处理的，不是继续补零散内容，而是统一改写写法。具体来说，论文要从“模块清单 + 功能说明 + 箭头流程”的表达方式，改成“提出问题、分析原因、给出方案、说明效果”的论述方式。只有先把写法改正，后续再决定是否增字，才不会出现字数增加了但论文气质仍像项目报告的问题。

此外，论文中已有若干处表述与代码实现不一致。如果这些问题不处理，答辩老师一旦追问，容易暴露“论文没有严格对应实际实现”的风险。因此，本轮修改建议应同时覆盖两条主线：

1. 论文写法改造。
2. 论文内容与代码实现对齐。

## 2. 对你列出问题的逐条展开

### 2.1 关于“组织结构要改成论述结构，而非提纲结构”

这个问题最明显地出现在第三章、第五章和第六章。

`thesis/md/chapter3_design.md` 中大量使用了“类设计：”“关键设计决策：”“协议概览：”“前端状态机：”这类提示词，读起来像设计文档，而不是毕业论文。例如 3.3 节目前基本是“模块名 + 两句总结”的并列堆叠，缺少以下三类学术写作动作：

1. 为什么要这样设计。
2. 这样设计解决了什么矛盾。
3. 这种设计相较替代方案的合理性是什么。

`thesis/md/chapter5_optimization.md` 的问题更典型。5.1 与 5.2 基本采用“问题描述 / 方案实现 / 效果”模板，这种格式便于做项目汇报，但连续六小节都这样写，会显得机械、提纲化、答辩提词卡化，缺少章节内部的自然过渡和推理展开。

`thesis/md/chapter6_testing.md` 中也存在相同问题。例如“pytest 运行摘要：”“核心指标取：”后直接列条目，说明性文字不足，导致测试章更像实验记录，不像结果分析。

建议修改原则如下：

1. 少用“冒号引出一串结论”的表达，改为完整句群。
2. 少用“并列列点”直接替代分析，改为“先总述，再分述，再归纳”。
3. 每节至少形成一个完整论证闭环，即“提出需求或问题 -> 说明设计或方法 -> 分析原因 -> 解释效果”。

可直接采用的改写方式如下。

原写法：

```text
关键设计决策：采用文本缓冲策略——LLM 产生的 token 先进入缓冲区，遇到句号、问号等标点或累积超过 15 字时整批送入 TTS，在合成质量与响应延迟间取得平衡。
```

建议改写为：

```text
在语音问答场景下，如果将大语言模型输出的每个 token 立即送入语音合成模块，虽然可以进一步压缩首帧等待时间，但会导致语音片段过短、韵律破碎，影响播报自然度；如果等待完整回答后再统一合成，又会抵消流式架构带来的实时性优势。基于这一矛盾，本文在 LLM 与 TTS 之间设计了文本缓冲机制：当缓冲文本达到一定长度，或检测到句号、问号等明显停顿标记时，再将其送入 TTS。该策略在响应速度与播报质量之间取得了较好的平衡。
```

### 2.2 关于“用例要改成表格，不要用冒号箭头，要论述”

这个问题集中出现在 `thesis/md/chapter3_design.md` 的 3.1.3。

当前写法如下：

- 用例 1、2、3、4 采用“参与者：”“前置条件：”“基本流程：”的固定模板。
- 流程内部大量使用 `->` 或 `→`。
- 四个用例本质上是“需求规格说明”的写法，而不是毕业论文中的“用例分析”写法。

建议改为“表格 + 段落分析”的组合形式，不要只保留表格。

推荐结构：

1. 先用一段话解释为什么本项目需要用例分析。
2. 给出“主要参与者与核心用例表”。
3. 表后分别对“语音问答”“多轮追问”“知识库管理”进行简短论述。

推荐表格字段如下：

| 用例编号 | 用例名称 | 参与者 | 触发条件 | 目标结果 |
|---|---|---|---|---|
| UC-1 | 语音问答 | 维修技术人员 | 用户开始录音并提出问题 | 系统返回可播报答案及来源 |
| UC-2 | 文本问答 | 维修技术人员 | 用户输入文本问题 | 系统返回文本答案及来源 |
| UC-3 | 多轮追问 | 维修技术人员 | 已存在上轮对话上下文 | 系统结合历史完成指代消解 |
| UC-4 | 知识库更新 | 系统管理员 | 有新增维修手册需要导入 | 文本与索引完成重建 |

然后在表后补论述，不要让表直接结束本节。例如：

```text
从表 3.x 可以看出，本项目虽然面向维修场景，但交互核心并不在于复杂业务流转，而在于围绕“提问、检索、生成、播报、打断”这一闭环构建稳定的实时问答流程。其中，语音问答是最主要的核心用例，多轮追问则体现了系统对上下文语义连续性的支持能力，知识库更新用例则保证了项目在工程落地时具备持续维护能力。
```

### 2.3 关于“单个表不能构成一个小节，必须要有论述”

目前第二章、第四章、第六章都存在这个问题。

典型位置包括但不限于：

1. `thesis/md/chapter2_technologies.md` 的 2.1.2、2.1.3、2.1.4、2.2.1、2.3.1、2.3.2、2.3.3。
2. `thesis/md/chapter4_implementation.md` 的 4.1.1。
3. `thesis/md/chapter6_testing.md` 的 6.1.1、6.3.3、6.4.3、6.5.1。

这些位置的共性是：

1. 表前只有一句“如表 x.x 所示”。
2. 表后只有一句“本项目选用……”或“测试项如表 x.x 所示”。
3. 表格承担了大部分信息量，正文解释严重不足。

这会让老师感觉“作者把分析工作交给了表”，而不是自己完成了学术论述。

建议统一改成“三段式”：

1. 表前交代比较维度或统计目的。
2. 表后分析表中最关键的 2 到 3 个结论。
3. 最后再落到本项目为何选择某方案。

例如 2.1.3 向量库对比，不能只写“对比见表 2.2”然后直接说“本项目选用 FAISS”，而应补出：

1. 为什么当前知识规模更适合单机方案。
2. 为什么不选择 Milvus 这类分布式方案。
3. 为什么原型验证阶段重视部署成本和可控性。

例如 6.4.3 实验结果表后，除了报告指标，还应解释：

1. 为什么 hybrid 的提升主要来自术语精确匹配。
2. 为什么 rerank 对 MRR 提升最明显。
3. 为什么在 1478 个片段规模下该提升仍有意义。

### 2.4 关于“缩略语、函数名、很多不明指代，可以多用类图”

这个问题本质上包括两类。

第一类是术语密度过高。当前论文中 STT、TTS、RAG、LLM、VAD、RRF、MRR、nDCG、BM25、SSE、SDK、PCM、API、OCR 等缩略语出现频繁，虽然很多地方首次出现时给了全称，但后文仍然存在“连续多个缩写堆叠”的情况，阅读门槛偏高。

第二类是代码标识符直接进入正文。当前第三章、第四章中出现了较多函数名、类名和消息名，例如 `call_soon_threadsafe`、`run_coroutine_threadsafe`、`process_query()`、`streaming_complete()`、`stt_partial`、`llm_chunk` 等。对于写代码的人来说这些名字很清楚，但对于论文评阅老师而言，它们只是实现细节，如果没有结构图配合，就会显得零散。

建议新增三类支撑内容：

1. 缩略语表。
2. 后端核心类关系图。
3. WebSocket 会话协作图或类图。

建议新增图如下：

1. 图 3.4 后端核心类关系图  
   展示 `VoiceChatPipeline`、`DocumentStore`、`StreamingGenerator`、`StreamingSynthesizer`、`StreamingRecognizer`、`AudioBuffer` 的关系。
2. 图 4.4 检索模块类协作图  
   展示 `DocumentStore` 如何组合 `EmbeddingClient`、`BM25Index`、`Reranker`。
3. 图 4.5 语音交互模块类图  
   展示 `NlsTokenManager`、`StreamingRecognizer`、`StreamingSynthesizer`、`_TtsCallback`、`AudioBuffer` 的交互边界。

类图的作用不是“让论文更像 UML 作业”，而是降低以下几种表述的理解成本：

1. “谁负责状态管理，谁负责回调转发”。
2. “为什么要线程隔离”。
3. “为什么某个类是组合关系而不是继承关系”。

同时建议在正文中减少裸露函数名的频率。原则是：

1. 关键函数名首次出现时可保留，用于体现实现真实性。
2. 后续再提及时，尽量改成“线程安全回调接口”“查询处理主流程”“音频合成结束调用”等中文描述。

### 2.5 关于“本系统措辞不准确，可以改成本项目；语音问答这个项目系统性不强”

这个意见成立，而且不只是措辞问题，而是论文定位问题。

当前“本系统”在论文中出现频繁，但你的工作更准确地说，是“面向航空维修场景的知识增强语音问答项目”或“语音问答项目实现方案”。如果全文一直写“本系统”，会有两个隐患：

1. 容易让表达显得空泛、模板化。
2. 当论文实际强调的是“项目设计与实现”时，“系统”一词会显得过大、过泛。

建议采用以下替换原则：

1. 在摘要、引言、结论等宏观位置，优先使用“本文设计并实现的语音问答项目”或“本项目”。
2. 在架构、模块、流程等技术描述位置，使用“该项目的后端服务”“项目中的检索模块”“语音问答流程”等更具体的说法。
3. 只有在确实强调“整体协同结构”时，再使用“系统”。

例如：

- “本系统选用 FAISS”改为“结合本项目当前知识规模与部署方式，本文选用 FAISS 作为向量检索组件”。
- “本系统采用 WebSocket”改为“该项目采用 WebSocket 作为实时双向通信机制”。
- “本系统在检索准确性方面达到了预期目标”改为“实验结果表明，本项目在检索准确性方面达到了预期目标”。

### 2.6 关于“字数增加到 2.2w 左右”

这里需要先明确一个事实：从 `thesis/md/*.md` 的粗略字符统计来看，全文字符量并不低，`wc -m` 统计总量约为 36255 字符，但这不等于 Word 最终统计口径下的正文“字数”。Markdown 标题、代码块、表格标记、英文缩写、空格和路径名都会影响这个数字，所以不能直接把当前字符数当成最终论文净字数。

因此，不建议现在机械地“为了凑 2.2w 而加字”。更合理的做法是：

1. 先把提纲式内容改写成论述性文字。
2. 删除明显不适合作为论文主体的代码式表达。
3. 再在三类位置补充真正有价值的论述。

最适合增字的位置如下：

1. 第一章研究意义与研究现状后的总结评述。
2. 第二章各技术方案对比后的选型理由分析。
3. 第三章需求与设计之间的过渡说明。
4. 第五章每项优化的原因分析和适用边界。
5. 第六章实验结果分析与讨论。
6. 第七章不足分析与未来展望。

不建议用来增字的位置如下：

1. 大段代码。
2. 大量接口字段清单。
3. 重复描述每个消息类型。
4. 重复介绍已在第二章讲过的技术概念。

换句话说，应该“增加分析性文字”，而不是“增加说明书式文字”。

## 3.5 论文与代码实现存在若干不一致

这部分必须优先修正，因为它比行文风格问题更容易在答辩中被指出。

### 3.5.1 文件名不一致

1. 论文写的是 `src/rag/document_loader_v2.py`，见 `thesis/md/chapter4_implementation.md` 第 70 行。
2. 仓库实际文件是 `src/rag/document_loader.py`。

### 3.5.2 检索实现描述不一致

论文 `thesis/md/chapter4_implementation.md` 第 78 行写的是“归一化后按 0.7:0.3 加权合并，取 Top-10 提交 `gte-rerank`，返回 Top-3 给 LLM`”。

实际代码在 `src/rag/retriever.py` 中：

1. 混合检索采用的是 `reciprocal_rank_fusion()`，不是 0.7:0.3 加权融合。
2. `search()` 的 `top_k` 默认是 5。
3. `fetch_k = top_k * 4`，也就是默认最多取 20 个候选，而不是固定 Top-10。
4. `pipeline.controller.py` 中调用 `self.rag.search(rewritten_query, top_k=5)`，因此返回给 LLM 的也是 Top-5 上下文，而不是 Top-3。

### 3.5.3 WebSocket 消息名不一致

论文 `thesis/md/chapter4_implementation.md` 第 126 行写到的下行消息为 `partial_transcript`、`final_transcript`、`sources`、`done`，并称消息带 `query_id`。

实际 `src/server/app.py` 中可确认：

1. STT 相关消息为 `stt_partial`、`stt_final`。
2. 来源消息为 `rag_sources`。
3. 结束消息包含 `llm_done`、`tts_done`、`tts_interrupted`、`history_cleared` 等。
4. STT 消息携带的是 `session_id`，而不是 `query_id`。
5. `llm_chunk` 和 `tts_audio` 并未统一携带 `query_id`。

这说明论文当前对通信协议的描述并不准确，必须按照真实实现重写。

### 3.5.4 Token 刷新机制表述不一致

论文 `thesis/md/chapter4_implementation.md` 第 84 行写的是“剩余不足 60 秒即调用 `_refresh_token()` 刷新”。

实际 `src/stt/recognizer.py` 中 `NlsTokenManager` 只有 `get_token()`，刷新逻辑内置在 `get_token()` 内部，没有名为 `_refresh_token()` 的公开或私有方法。这里应改成“在 `get_token()` 内部判断过期时间并按需重新获取 Token”。

### 3.5.5 TTS 相关表述不一致

论文 `thesis/md/chapter4_implementation.md` 第 104 行写的是“输出 16 kHz PCM”。

实际 `src/tts/synthesizer.py` 中使用的是 `AudioFormat.PCM_22050HZ_MONO_16BIT`，也就是 22050 Hz 单声道 16 位 PCM。这里如果前端播放时有重采样，应在论文里交代清楚；如果没有，就不能写成 16 kHz。

另外，论文第 108 行给出的函数名是 `_preprocess()`，实际实现名为 `_preprocess_for_tts()`。

### 3.5.6 检索模块文件名与索引文件名不一致

论文 `thesis/md/chapter4_implementation.md` 第 78 行写“在线检索在 `src/rag/document_store.py` 完成”，但仓库中实际实现位于 `src/rag/retriever.py`，核心类为 `DocumentStore`。

论文第 74 行写“保存为 `faiss.index`、`bm25.pkl`、`docs.json`”，实际代码保存的是：

1. `index.faiss`
2. `documents.json`
3. `bm25_index.pkl`
4. `bm25_corpus.json`
5. `bm25_documents.json`

### 3.5.7 Pipeline 代码片段与真实类成员不一致

论文 `thesis/md/chapter4_implementation.md` 第 132 至 139 行中的示例代码使用了 `self.generator`、`self.synthesizer`。

但实际 `src/pipeline/controller.py` 中成员名是：

1. `self.llm`
2. `self.tts`

这类细节如果进入论文，就必须与真实代码保持一致，否则建议改为伪代码或流程描述，避免被逐字核对。

## 4. 分章节修改建议

## 4.1 摘要与题目

摘要整体信息量足够，但建议做两类调整。

第一类是定位调整。题目和摘要中“Agent 系统”可以改得更稳妥一些，例如：

1. 面向航空维修知识增强的语音问答项目设计与优化
2. 面向航空维修领域的知识增强语音问答项目设计与实现

第二类是措辞统一。摘要中建议少用“本系统”，多用“本文实现的语音问答项目”或“本项目”。

此外，摘要不必过度强调前端“航空仪表盘风格”，这类表述更适合项目展示，不是论文摘要最该突出的贡献点。摘要重点应放在：

1. 面向航空维修场景。
2. RAG 与流式语音协同。
3. 检索指标提升。
4. 端到端延迟降低。

## 4.2 第一章引言

第一章总体框架没有大问题，但还可以增强“评述性”，减少“事实堆叠”。

建议重点修改：

1. 1.2 研究现状每个小节最后补一句评述，说明现有研究对你这个项目仍有哪些不足。
2. 1.3 本文主要工作目前是五个大点，建议每一点扩写为“工作内容 + 解决的问题 + 结果”三句式。
3. 1.4 组织结构不要逐章机械介绍，可改为一段总述加一句突出第三至第六章的逻辑关系。

## 4.3 第二章相关技术

第二章目前已经比早期“纯技术百科”版本好一些，但仍然有“表格驱动正文”的倾向。

建议按以下方式继续强化：

1. 每个技术小节都保留“概念 -> 对比 -> 选型”，但要把“选型理由”扩成完整论述。
2. 不要只写“本项目选用 FAISS / Qwen-Plus / NLS / CosyVoice”，而要写“为什么它比备选更适合当前项目边界”。
3. 对于 2.3.3，建议把 WebSocket 和并发模型的关系讲清楚，即“协议层解决的是双向传输问题，并发模型解决的是后端执行问题”，避免两者并列却逻辑上断开。

第二章还建议增加一个“术语与缩略语约定”小表，可放在 2.1 前或 2.4 小结前，列出文中最常用缩略语，降低后文阅读门槛。

## 4.4 第三章系统需求分析与设计

第三章是当前最需要重写的一章之一。

### 4.4.1 3.1 需求分析

1. 将 3.1.1、3.1.2 中的五个分点保留，但每一点都要从“列表句”改成“说明句”。
2. 3.1.3 用例分析必须改成“图 + 表 + 段落论述”。
3. 用例节最后补一段总结，说明这些用例如何映射到后文的模块设计。

### 4.4.2 3.2 架构设计

1. 图 3.2 与图 3.3 保留。
2. 每张图后都要写“图示信息说明 + 设计原因 + 对后文实现的约束”。
3. 3.2.3 线程与协程模型可以配一个小类图或职责图，而不是只靠函数名解释。

### 4.4.3 3.3 模块设计

这里不建议继续使用“类设计 / 关键设计决策”的模板。建议改成下面这种结构：

1. 先说明模块职责。
2. 再说明模块内部核心对象及其关系。
3. 最后说明关键设计取舍。

例如 3.3.2 RAG 模块可以这样写：

1. 模块职责：负责从知识库中召回与问题最相关的技术片段。
2. 核心对象：`DocumentStore`、`EmbeddingClient`、`BM25Index`、`Reranker`、`QueryRewriter`。
3. 设计取舍：为何采用“改写 + 混合召回 + 重排序”的串联流程。

也就是说，类名可以保留，但必须嵌入解释，而不是单独作为标签列出。

### 4.4.4 3.4 数据设计

当前直接贴 JSON 结构，偏工程文档。建议改为：

1. 先用文字介绍三类数据对象的作用。
2. 再保留简化后的结构示意。
3. 结构示意不要过多保留 JSON 标点，可以改成表格或字段说明。

论文中可以出现结构示意，但不应让大量 JSON 形式占据主体篇幅。

## 4.5 第四章系统实现

第四章是另一个需要重点修正的章节，原因不是内容不够，而是“实现叙述”和“真实代码一致性”两方面都要调整。

### 4.5.1 4.1 总体实现

1. 4.1.1 不建议连续放三个表后只写一句话，建议压缩表格数量，并在表后说明“这些环境选择如何支撑实时语音问答任务”。
2. 4.1.2 至 4.1.4 基本结构合理，但每张图后最好补一段“图中最关键的设计关系”。

### 4.5.2 4.2 RAG 模块实现

1. 先修正文件名、索引文件名和检索融合策略的不一致。
2. 文档预处理要从“工具清单说明”改成“为什么航空维修手册必须采用版面分析 + OCR”的问题导向写法。
3. 在线检索不能再写 0.7:0.3 加权融合，必须按真实代码改成 RRF。

### 4.5.3 4.3 至 4.5 三个核心模块

这三节内容基础是够的，但建议统一写法：

1. 模块运行约束是什么。
2. 为什么采用当前实现方式。
3. 关键实现怎样解决约束。

例如 STT 节要强调“第三方 SDK 阻塞特性与 asyncio 事件循环的冲突”；TTS 节要强调“回调线程与 WebSocket 发送线程的边界管理”。

### 4.5.4 4.6 Pipeline 与消息路由

当前章节标题是“Pipeline 编排模块实现”，但 4.6.1 实际又写成了“WebSocket 消息路由与连接管理”，容易造成读者误解。

建议做法：

1. 将 4.6 标题改为“会话编排与消息协同实现”。
2. 4.6.1 写“会话连接与消息路由”。
3. 4.6.2 写“查询处理主流程”。
4. 4.6.3 写“文本缓冲与打断控制”。

这样标题与内容更一致。

同时，第四章中的代码片段建议控制数量。只保留 1 至 2 段最能体现设计亮点的代码，其他地方尽量改成伪代码或文字分析，否则会继续强化“项目报告感”。

## 4.6 第五章系统优化

第五章当前信息很多，但写法过于模板化，是需要重点重写的章节。

建议不要再连续使用“问题描述 / 方案实现 / 效果”三级标签。可以改成以下两种方式交替使用：

1. 对于 RAG 优化，采用“问题背景 -> 方法设计 -> 指标变化 -> 原因分析”。
2. 对于流式优化，采用“性能瓶颈 -> 改进措施 -> 延迟影响 -> 适用边界”。

特别需要补强的是“原因分析”和“适用边界”。

例如：

1. 混合检索为什么在航空维修术语上有效。
2. 查询改写在什么情况下有效，什么情况下可能不值得调用。
3. 预缓冲为什么能提升体验，但为什么不能无限增大缓冲时长。
4. 打断机制为什么要采用多层协同，而不是单一停止信号。

第五章实际上很适合增加高质量字数，因为这里最容易写出“分析性文字”，而不是水字数。

## 4.7 第六章系统测试与结果分析

第六章已经有数据基础，但“分析”还不够充分。

建议重点补充以下内容：

1. 6.2 单元测试不要只写“多少个用例通过”，还要说明这些测试覆盖了哪些关键风险。
2. 6.3 端到端测试除了列“通过”，还应说明这些测试为何能覆盖主流程。
3. 6.4 结果分析还可以再展开一段，专门解释为什么 MRR 的提升比 Hit Rate 更能说明 rerank 的价值。
4. 6.5 延迟分析可增加一句“延迟构成中哪一项最稳定，哪一项最受云服务波动影响”。

另外，第六章很适合加入“局限性讨论”。例如：

1. 45 条测试查询规模仍然偏小。
2. 音频测试数据以合成音频为主，真实现场噪声仍需进一步验证。
3. 云端 API 时延波动可能影响实验复现性。

这样的讨论会显著提升论文的学术成熟度。

## 4.8 第七章结论与展望

第七章当前结构清楚，但语言略偏总结清单。

建议做两点增强：

1. 7.1 工作总结不要只复述做了什么，要强调“本文工作的核心贡献在于把知识检索和流式语音交互统一到同一项目框架中”。
2. 7.2 与 7.3 可以写得更像“研究反思”，而不是简单列出后续计划。

例如未来展望中可以区分：

1. 工程扩展方向。
2. 算法优化方向。
3. 应用验证方向。

## 6. 建议优先级

如果按修改收益排序，我建议按下面顺序处理。

1. 先修正所有“论文与代码不一致”的内容。
2. 再重写第三章 3.1.3 和 3.3，解决用例与模块设计的提纲化问题。
3. 再重写第五章，把“问题描述 / 方案 / 效果”改成论述型写法。
4. 再补强第二章和第六章表后分析。
5. 最后统一全文措辞，把“本系统”替换成“本项目”或更具体的表达。
6. 完成上述改动后，再用 Word 的真实统计口径核定是否需要补到 2.2 万字左右。
