# Bug 修复方案：代码审查问题汇总（2026-04-01）

## 背景

本轮代码审查发现，当前主链路存在若干会直接影响语音问答稳定性的缺陷，主要集中在以下几个方面：

1. STT 最终结果重复上报，导致同一问题被执行两次
2. 录音停止与模式切换时，旧会话结果可能串入当前会话
3. 自动打断后，新问题可能被 `processing` 门闩直接吞掉
4. RAG 长段落切分逻辑会丢句、重复句
5. NLS token 获取逻辑存在敏感信息泄露和异常处理不完整问题

这些问题优先级高于体验类优化，应先修复主链路正确性。

---

## 问题 1：STT 最终结果重复触发查询

### 现象

同一段语音在部分时序下会被处理两次，表现为：

- 前端收到两次 `stt_final`
- 后端执行两次 `process_query`
- AI 重复回答同一个问题

### 根因分析

后端当前有两条路径都会触发最终文本处理：

1. NLS SDK 在 `on_sentence_end` 中回调最终文本
2. 前端发送 `stop_recording` 后，后端再次调用 `stop()` 并返回 `_final_text`

当前实现中这两条路径都会：

- 发送 `stt_final`
- 调度 `process_query`

因此存在重复触发。

### 修复方案

为每个 `StreamingRecognizer` 增加“一次性最终结果提交”机制：

- SDK 回调到达时只允许提交一次
- `stop()` 返回时如果最终结果已经提交过，则不再重复提交
- `stop_recording` 路径与 SDK 回调路径统一走同一套去重逻辑

### 涉及文件

- `src/stt/recognizer.py`
- `src/server/app.py`

## 问题 2：旧 STT 会话结果串扰当前会话

### 现象

模式切换、快速重新录音或旧 STT 连接延迟关闭时，旧会话返回的 `stt_final` 可能落入当前前端状态机，导致：

- 用户界面展示旧文本
- 当前轮对话被错误推进
- `discard=true` 场景下仍可能触发查询

### 根因分析

虽然当前引入了 `session_id`，但并未完整贯穿：

- `start_recording` 创建的回调会携带 `session_id`
- 但 `stop_recording` 路径发送的 `stt_final` 没有 `session_id`
- 后端也没有在提交最终结果前校验当前 STT 是否仍是活动实例

这会导致旧识别器晚到的结果仍然被接收和处理。

### 修复方案

引入后端活动会话校验：

- WebSocket 作用域内维护当前活动 `session_id`
- 每个识别器的最终提交前都校验该 `session_id` 是否仍然有效
- `stop_recording` 路径发出的 `stt_final` 也必须带上 `session_id`
- `discard=true` 时禁止提交最终文本和查询

### 涉及文件

- `src/server/app.py`
- `src/stt/recognizer.py`

## 问题 3：自动打断后的新问题被静默丢弃

### 现象

持续监听模式下，用户在 AI 播报时再次说话，前端会自动发送 `interrupt`。如果新一句的 `stt_final` 到达时，旧请求尚未完全退出，则新的查询可能被直接丢弃，表现为：

- 前端已显示识别文本
- 但没有后续回答
- 没有错误提示

### 根因分析

后端当前使用布尔变量 `processing` 做互斥：

```python
if processing:
    return
```

这会把新请求静默丢掉，而不是等待旧请求退出后继续执行。对于“打断后立刻接新问题”的交互，这是错误语义。

### 修复方案

将布尔门闩改为串行执行锁：

- 使用 `asyncio.Lock()` 保证同一连接内一次只执行一个查询
- 查询在拿到锁后再次检查 `generation`
- 如果请求在等待锁期间已经过期，则直接跳过
- 新问题不会因为旧请求尚未结束而被吞掉，只会串行接续

### 涉及文件

- `src/server/app.py`

## 问题 4：RAG 长段落切分会丢句、重复句

### 现象

当段落长度超过 `max_chunk_size` 时，当前切句逻辑可能出现：

- 最后一段被丢失
- 前一句被重复
- 构建出来的向量索引内容失真

### 根因分析

`re.split(r'([。！？])', text)` 之后的句子重组逻辑错误地按奇偶长度补句，导致尾句处理失真。

示例：

```python
text = "你好。世界"
```

当前逻辑会生成：

```python
["你好。", "你好。"]
```

正确结果应包含尾句 `世界`。

### 修复方案

重写长段落切句逻辑：

- 按“句子内容 + 可选标点”稳定重组
- 保留无终止标点的尾句
- 再按 `max_chunk_size` 合并为 chunk

### 涉及文件

- `src/rag/document_loader_v2.py`

## 问题 5：NLS token 获取存在泄露与异常处理缺陷

### 现象

当前 token 获取逻辑存在两个问题：

1. 直接 `print` token、响应体和过期时间
2. 失败时仅打印异常，不抛出，调用方可能继续以无效 token 启动 STT

### 风险

- 敏感 token 暴露到控制台和日志
- 上层收到模糊错误，排查困难

### 修复方案

- 移除所有敏感 `print`
- 统一改为结构化日志
- token 拉取失败时显式抛出异常
- 上层保留原有错误回传逻辑

### 涉及文件

- `src/stt/recognizer.py`
